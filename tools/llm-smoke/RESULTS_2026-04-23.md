Headless `tools/llm-smoke/eval.py` run on M4 Max, MLX mlx-lm HEAD, 4-bit
weights from `mlx-community`. Scenarios transplanted from
`ios/MCPZimEval/EvalHarness.swift`; every prompt exercises tool
selection (name match + required-arg substring match), not the end-user
answer body ‚Äî that requires a live ZIM/streetzim and is measured
separately by the Swift harness.

## Common miss: `restaurants_in_sf`

Every model except Qwen picked `near_places` (radius around user) instead
of `near_named_place` (radius around named location). The tool
descriptions are similar; the prompt "in San Francisco" reads as a
location anchor but could also be user-context. Tightening the
`near_places` description to "user is already there" would likely lift
the score for both Gemma and Phi.

## Phone-budget analysis

Phone (iPhone 17 Pro Max, 6 GB jetsam) budget, from
`QWEN35_HYBRID_CACHE.md`:
- Qwen 3 4B 4-bit: 2.0 GB weights + ~600 MB overhead = ~2.6 GB active
- 7 k-token preamble KV cache at 4-bit: ~400 MB
- Leaves ~3 GB for MapLibre / WKWebView / Metal scratch

| Candidate | Fits phone? | Cache reuse? |
|---|---|---|
| gemma-3-4b-it-4bit | yes (2.2 GB ‚âà 2.8 GB active) | **yes ‚Äî dense attention**; no MambaCache bug like Qwen 3.5 |
| Qwen3-4B-Instruct-2507-4bit | yes | yes (current) |
| Phi-4-mini-instruct-6bit | **tight** (3.1 GB ‚âà 3.7 GB active) | yes (dense) |
| Phi-4-mini-instruct-4bit | yes ‚Äî but tool-calling is broken |
| Nemotron variants | ‚Äî (tool-calling unusable) |

## Recommendation

**Top candidate to displace Qwen 3 4B Instruct is gemma-3-4b-it-4bit**:
same phone budget, +1 scenario on the LLM-only eval, same ~70 tok/s
decode, dense attention (no MambaCache reuse bug).

Before swapping the default:

1. **Rerun the Swift `EvalHarness` on iPhone hardware** ‚Äî our app has the
   IntentRouter fast-path for `restaurants_in_sf` / `compare_*`, plus a
   longer system preamble with stricter tool hints. Real app scores
   often run 1‚Äì2 points higher than the LLM-only harness, and the
   tie-breakers shift.
2. **Verify `MLXLLM.Gemma3Text` loads through `Gemma4Provider`'s pipeline
   or needs a sibling `Gemma3Provider`** ‚Äî gemma-3 uses `<start_of_turn>`
   / `<end_of_turn>` markers and different tool-call conventions, so a
   template adapter is required either way.
3. **Probe for cache reuse correctness** ‚Äî `tools/gemma-smoke
   cache-probe` against gemma-3 to confirm BPE stability + no stale
   per-eval state on the model class (the `precomputedPositionIds` bug
   that killed Qwen 3.5 is model-specific; gemma-3 has its own scratch
   fields ‚Äî mlx-swift-lm#157 lists `GlmOcr` and `Qwen3VL` but not
   `Gemma3Text`).

## What's out

- **Phi family**: 4-bit quant damages tool-calling; 6-bit fits neither
  the phone memory budget nor the decode-speed target.
- **Nemotron 4B**: both v1.1 (Llama-3.1) and v3 ignore our tool
  schemas almost entirely. Would need finetuning or prompt
  restructuring not worth pursuing.
- **Qwen 3 Thinking**: scratchpad dominates output; even with `<think>`
  stripping, decode at 39 tok/s is too slow for the app's latency
  target.

## Harness notes

Run any model:

```bash
cd tools/llm-smoke
source .venv/bin/activate
python eval.py mlx-community/<model-id>           # all 9
python eval.py mlx-community/<model-id> one --show-prompt
```

Harness auto-injects tool schemas two ways:
1. `tokenizer.apply_chat_template(tools=...)` if the family supports it
   (Qwen, Gemma 3, Llama 3).
2. `message['tools']` direct attachment ‚Äî required for Phi-4-mini which
   reads `message['tools']` from the jinja body rather than the kwarg.
3. Falls back to an in-system `<tool_call>`-style envelope for
   templates that ignore both (none in this round).
