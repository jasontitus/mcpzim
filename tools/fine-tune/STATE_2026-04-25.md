## Failure modes seen + fixes

| Symptom | Cause | Fix |
|---|---|---|
| Gemma 3 stays on CPU even with `device_map="auto"` | transformers 5.x + accelerate 1.x interaction | Force `.to("cuda")` (already done in `finetune_cuda.py`) |
| `torch 2.X.X+cpu` after llama-cpp-python rebuild | uv resolver swapped CUDA wheel for CPU during deps re-pin | Re-install `torch` with `--index-url https://download.pytorch.org/whl/cu121` |
| `cmake: command not found` during llama.cpp build | system cmake not installed on WSL box | Use `~/mcpzim-ft/.venv/bin/cmake` (uv pip install cmake ninja) |
| `libnvJitLink.so.13: cannot open shared object` from bitsandbytes | benign — bitsandbytes detects CUDA 13 lib mismatch | Ignore; training proceeds normally |
| Gemma 3 ~7× slower than Qwen at same params | SWA + SDPA + grad-checkpoint heavy interaction | Confirmed not VRAM thrashing (tested bsz=2 — same speed). Just slow; let it grind. |
| `RuntimeError: ... out of memory` on Gemma 4B | LM Studio still using VRAM, or bsz too high | Confirm LM Studio killed (`taskkill.exe /F /IM "LM Studio.exe"`); drop bsz to 1 |

## Mac Studio MLX fallback

If pcgaming is unreachable for some reason, Mac Studio can run the **original** MLX pipeline:

```sh
cd tools/fine-tune
pmset displaysleepnow                                  # relax Metal interactivity watchdog
bash train_all.sh train_v4_combined.jsonl              # mlx-lm pipeline, 4 candidates
```

The display-off step is critical. M2 Max hit Metal "Impacting Interactivity" GPU watchdog kills mid-training; Mac Studio has more thermal headroom but the same kernel rules. See commits 4d34fcf and the surrounding `finetune.sh` history for the failure mode.
